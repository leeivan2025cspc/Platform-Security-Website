Lesson 5: Platform Architecture & Trusted Platform Module (TPM)

Understanding Platform Architecture

Platform architecture defines how the different layers of a computing system work together, from the physical hardware to the applications we use every day. Each layer plays a crucial role in maintaining functionality, efficiency, and most importantly, security.

Layers of a Computing Platform

  • Hardware: Physical components like CPU, memory, and storage form the foundation. Security risks at this level include physical tampering and side-channel attacks.
  • Firmware: Programs like BIOS or UEFI that initialize hardware. Attackers may implant firmware-level malware that hides from the OS.
  • Operating System: Acts as the bridge between users, applications, and hardware. Common threats include privilege escalation and kernel exploits.
  • Applications: User-level software. Vulnerabilities in apps can expose the entire platform to malware or data breaches.

Real-world example: The 2018 Meltdown and Spectre vulnerabilities exploited flaws in CPU architecture, showing that even hardware design decisions can affect global cybersecurity.

Mobile vs Desktop Platforms

Different platforms have unique architectures and therefore different security priorities.

Mobile Platforms

  • Highly integrated hardware and software.
  • App sandboxing limits data access between apps.
  • Frequent updates and permissions for tighter control.

Desktop Platforms

  • Offer flexibility and performance at the cost of a larger attack surface.
  • Users often have administrative privileges, increasing risk.
  • Malware can run persistently if not properly managed.

Example: Android uses sandboxing to prevent apps from reading each other's data, while Windows prioritizes user control and versatilityâ€”sometimes at the cost of stricter isolation.

Trusted Platform Module (TPM)

The Trusted Platform Module is a dedicated hardware chip designed to enhance security by providing cryptographic functions and secure key storage. It acts as a "root of trust" for the system.

  • Secure Storage: TPM keeps encryption keys, passwords, and certificates isolated from the main CPU and OS.
  • System Integrity: Verifies that firmware and the OS have not been tampered with before booting.
  • Hardware-based Encryption: Offers stronger protection than software-only solutions.

Chain of Trust

TPM verifies each stage of the boot process:

  • Firmware checks the bootloader.
  • Bootloader verifies the operating system kernel.
  • The OS validates drivers and critical components.

Real-world example: Windows BitLocker uses TPM to securely store disk encryption keys. Without TPM validation, BitLocker refuses to unlock the drive, preventing attackers from booting the OS on another device.

TPM Versions and Limitations

  • TPM 1.2: Introduced RSA and SHA-1 cryptography.
  • TPM 2.0: Supports stronger algorithms like SHA-256 and ECC, required by Windows 11.

Benefits of TPM

  • Prevents firmware-level malware (rootkits and bootkits).
  • Ensures secure boot processes.
  • Supports trusted computing in enterprise environments.

Limitations

  • Not all devices have TPM hardware.
  • Failed TPM chips can cause data loss on encrypted drives.
  • Cannot protect against OS-level malware or physical attacks once booted.

Example: Many modern laptops now ship with integrated TPM 2.0 chips, while older PCs require firmware-based fTPM solutions from AMD or Intel. This shift shows how hardware trust anchors are becoming standard in modern computing.
